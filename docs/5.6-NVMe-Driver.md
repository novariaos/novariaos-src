# NVMe Driver

The NVMe driver implements a PCIe NVMe storage controller supporting read and write operations with interrupt-based I/O via MSI.

## Overview

NVMe (Non-Volatile Memory Express) is a host controller interface for SSDs connected over PCIe. Unlike ATA, NVMe uses paired submission/completion queues and supports multiple namespaces on a single controller.

The driver:
- discovers the controller via PCI class code `01:08:02`
- maps the MMIO register space from BAR0
- resets and enables the controller
- allocates page-aligned admin and I/O queues via the buddy allocator
- enumerates all active namespaces and registers a block device for each

## Initialization sequence

```
nvme_init()
  ├── pci_find_device_by_class(01:08:02)   — locate controller
  ├── read BAR0                             — MMIO base address
  ├── enable bus mastering (PCI CMD)
  ├── nvme_reset_controller()              — clear CC.EN, wait CSTS.RDY=0
  ├── nvme_setup_admin_queues()            — allocate ASQ/ACQ, write AQA/ASQ/ACQ
  ├── nvme_enable_controller()             — set CC, wait CSTS.RDY=1
  ├── nvme_identify_controller()           — get NN (number of namespaces)
  ├── nvme_setup_io_queues()               — create I/O CQ and SQ (admin commands)
  ├── idt_install_handler + pci_enable_msi — interrupt-based I/O
  └── for ns in 1..NN:
        nvme_identify_namespace(ns)        — get NSZE
        register_block_device("nvme0nN")  — if NSZE > 0
```

## MMIO register map

The controller registers are accessed via the MMIO BAR0 region:

| Offset     | Register  | Description                          |
|------------|-----------|--------------------------------------|
| `0x00`     | `CAP`     | Controller Capabilities (64-bit)     |
| `0x08`     | `VS`      | Version                              |
| `0x14`     | `CC`      | Controller Configuration             |
| `0x1C`     | `CSTS`    | Controller Status                    |
| `0x24`     | `AQA`     | Admin Queue Attributes               |
| `0x28`     | `ASQ`     | Admin Submission Queue Base Address  |
| `0x30`     | `ACQ`     | Admin Completion Queue Base Address  |
| `0x1000+`  | Doorbells | Submission/completion queue doorbells|

Doorbells begin at offset `0x1000`. For queue `n`, the submission doorbell is at `0x1000 + n*2*(1 << DSTRD)` and the completion doorbell at `0x1000 + (n*2+1)*(1 << DSTRD)`.

## Queues

The driver uses two queue pairs:

| Queue     | Size  | Purpose                              |
|-----------|-------|--------------------------------------|
| Admin SQ  | 64    | Submit admin commands (Identify, Create CQ/SQ) |
| Admin CQ  | 64    | Receive admin completions            |
| I/O SQ    | 256   | Submit read/write commands           |
| I/O CQ    | 256   | Receive I/O completions              |

All queue buffers are allocated directly from the buddy allocator to guarantee 4 KiB page alignment — a requirement of the NVMe specification. `kmalloc` cannot be used here because it prepends a 16-byte header that breaks alignment.

## DMA memory allocation

```c
static void* alloc_dma(size_t size) {
    size_t order = BUDDY_MIN_ORDER;
    size_t block_size = BUDDY_BLOCK_SIZE(order);
    while (block_size < size && order < BUDDY_MAX_ORDER) {
        order++;
        block_size <<= 1;
    }
    return buddy_alloc(slab_get_buddy(), block_size);
}
```

`buddy_alloc` returns blocks at naturally aligned addresses because the pool base is page-aligned and all blocks are power-of-2 sized.

## Command format

Each NVMe command is 64 bytes:

| Field    | Bits  | Description                          |
|----------|-------|--------------------------------------|
| `CDW0`   | 31:16 | Command ID                           |
| `CDW0`   | 7:0   | Opcode                               |
| `NSID`   | 31:0  | Namespace identifier                 |
| `PRP1`   | 63:0  | Physical address of data buffer      |
| `CDW10`  | 31:0  | LBA low (read/write) or CNS (identify)|
| `CDW11`  | 31:0  | LBA high (read/write)                |
| `CDW12`  | 15:0  | Number of blocks minus 1             |

## Namespace support

The driver calls Identify Controller (CNS=1) to read `NN` (Number of Namespaces, at byte offset 516 in the response). It then calls Identify Namespace (CNS=0) for each NSID `1..NN`. Namespaces with `NSZE=0` are skipped.

Each active namespace is registered as a block device named `nvme0nN` (e.g. `nvme0n1`, `nvme0n2`). Per-namespace state (`nsid`, `block_count`) is stored in `nvme_ns_ctx_t` and attached to the block device via `private_data`.

## Interrupts (MSI)

After I/O queues are created, the driver:

1. Installs `nvme_irq_handler` at IDT vector `0x41` via `idt_install_handler`
2. Programs MSI at LAPIC address `0xFEE00000`, data `0x41`
3. Clears LAPIC TPR and executes `sti`

The IRQ handler reads one completion entry from the I/O CQ, records the status in `nvme_io_result`, advances `io_cq_head`, rings the completion doorbell, and sends EOI. `nvme_submit_io_and_wait` then waits on `nvme_io_complete`.

## Admin commands used

| Opcode | Name           | Purpose                            |
|--------|----------------|------------------------------------|
| `0x06` | Identify       | Controller info, namespace info    |
| `0x05` | Create I/O CQ  | Create the I/O completion queue    |
| `0x01` | Create I/O SQ  | Create the I/O submission queue    |

## I/O commands used

| Opcode | Name  | Max blocks per command |
|--------|-------|------------------------|
| `0x02` | Read  | 256                    |
| `0x01` | Write | 256                    |

Transfers larger than 256 blocks are split into multiple commands automatically.

## Registered devices

| Name       | Driver | Block size | Writable |
|------------|--------|------------|----------|
| `nvme0n1`  | NVMe   | 512 B      | yes      |
| `nvme0n2`… | NVMe   | 512 B      | yes      |
